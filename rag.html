I. Indexing Offline · runs once per document
1
Load documents
pending
Raw source material is loaded into the pipeline — PDFs, wikis, HR docs, customer tickets, anything textual. In production this is the ingestion layer.
2
Chunk
pending
Long documents are split into smaller, semantically-coherent chunks. Chunk size and overlap are tuning parameters — too big and similarity gets noisy, too small and context is lost. Here each line is one chunk.
3
Embed chunks
pending
Each chunk is passed through an embedding model and converted into a dense vector — typically 768 to 3,072 dimensions. We use 8 dimensions here for visibility. Geometrically close vectors = semantically close meaning.
4
Store in vector database
pending
Vectors are persisted to a specialized store — Pinecone, Weaviate, pgvector, Chroma — indexed for fast nearest-neighbour search. This is where the indexing phase ends.
II. Retrieval & Generation Inference · runs on every query
5
Receive user query
pending
A user asks a question in natural language. Nothing happens to the documents at this point — the index is already built.
6
Embed the query
pending
The query goes through the same embedding model used for chunks — this is essential. If the encoders differ, the vector spaces don't align and similarity is meaningless.
7
Similarity search
pending
Cosine similarity is computed between the query vector and every chunk vector. Top-k results (k=3 here) are selected. This is the "Retrieval" in RAG.
8
Inject context into prompt
pending
Retrieved chunks are slotted into a prompt template alongside the user's question. This grounded prompt is what gets sent to the LLM — not just the bare query.
9
Generate the answer
pending
The LLM reads the grounded prompt and produces a response — drawing primarily from the injected context rather than its parametric memory. This is the "Generation" in RAG.
